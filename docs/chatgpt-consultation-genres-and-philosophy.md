## 相談事項1: GENRESを動員に絞る

### 現在のGENRES（8種類）

```typescript
export const GENRES = [
  { id: "idol", label: "アイドル", icon: "🎤" },
  { id: "vtuber", label: "VTuber", icon: "🎮" },
  { id: "artist", label: "アーティスト", icon: "🎸" },
  { id: "creator", label: "クリエイター", icon: "✏️" },
  { id: "actor", label: "俳優・女優", icon: "🎬" },
  { id: "voice_actor", label: "声優", icon: "🎙️" },
  { id: "streamer", label: "配信者", icon: "📺" },
  { id: "other", label: "その他", icon: "⭐" },
] as const;
```

### 問題点

**ジャンルが広がりすぎて、ユーザーが迷う**

- 「動員」という目的に対して、8種類のジャンルは多すぎる
- 「クリエイター」「その他」は曖昧で、何を選べばいいか分からない
- 「俳優・女優」「声優」はライブ動員のイメージが薄い

### 質問1: どのジャンルを残すべきか

**動員（ライブ動員）に適したジャンルはどれですか？**

候補案:
- **A案**: アイドル/VTuber/アーティストの3つのみ
- **B案**: アイドル/VTuber/アーティスト/配信者の4つ
- **C案**: アイドル/VTuber/アーティスト/声優の4つ
- **D案**: その他の組み合わせ

### 質問2: 既存データの扱い

**既存データ（俳優・女優、クリエイター等）はどう扱うべきですか？**

- **A案**: フォールバック表示（表示のみOK、新規作成不可）
- **B案**: 「その他」に統合
- **C案**: 削除（データ移行）

## 相談事項2: Don't Work!思想の統合

### Don't Work!とは

**思想**: Don't Work!（叩くな、没頭しろ）

**コンセプト**: 『没頭できることで飯を食う』という人間の望みを解決する

### AI時代のプロダクト開発の本質

> AI時代のプロダクト開発は、表に見える機能・画面ではなくて、
> 『思想』『前提』『ポリシー』で成果が決まる。

#### ①思想
→「このサービスは、誰のどんな幸せを増やすのか」という芯

#### ②前提
→「どんな状況を想定しているか」「ユーザーは何ができて、何ができないか」

#### ③ポリシー
→「守るべきルール」と「危ないことをしないための仕組み」

### 現在のプロジェクトの『思想』『前提』『ポリシー』

#### Phase 2実装ガイドに記載されている内容

**思想（Why）**:
- ユーザーは「ログイン」という言葉に不安を感じる
- 「迷わない／怖がらず／戻ってこれる」体験を提供する

**前提（Assumptions）**:
- ユーザーは専門家ではない
- まず「何ができるか」が一瞬で伝わる必要がある
- login / OAuth は 黒箱（触らない）

**ポリシー（Rules）**:
- DBを壊さない
- 既存データを消さない
- 危険な変更は diff-check で止める
- 機能追加は「案内の追加」から始める

### 質問3: Don't Work!思想をどう統合すべきか

**プロジェクト全体の『思想』『前提』『ポリシー』をどう明文化すべきですか？**

以下の観点で相談したいです：

1. **プロジェクトの芯（思想）**
   - 「君斗りんくの動員ちゃれんじ」は、誰のどんな幸せを増やすのか
   - Don't Work!（没頭できることで飯を食う）とどう繋がるのか

2. **ユーザー像（前提）**
   - 誰が使うのか（クリエイター？ファン？両方？）
   - どんな状況で使うのか
   - 何ができて、何ができないのか

3. **安全性（ポリシー）**
   - 法務的に安全な表現（「判断材料になる」「応援を可視化する」）
   - 誇張表現の禁止（「売れる」「集客できる」等）
   - データの扱い

### 質問4: 機能追加の判断基準

**Don't Work!思想に基づいて、機能追加をどう判断すべきですか？**

例えば、以下の機能は追加すべきか：
- YouTubeプレミア同時視聴（streaming）
- 作品リリース応援（release）
- グッズ・物販応援（goods）
- 調査・アンケート（survey）

判断基準:
- この機能は「没頭できることで飯を食う」に繋がるか
- この機能は「動員」という芯に合うか
- この機能は「ユーザーを迷わせない」か
