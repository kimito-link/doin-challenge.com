# Web版リリースチェックリスト（Gate 1〜4）

**プロジェクト:** 君斗りんくの動員ちゃれんじ

**最終更新:** 2026-01-27

---

## 📋 チェックリストの使い方

このチェックリストは**リリースゲート方式**で構成されています。各Gateを順番にクリアしてください。

- **Gate 1**: 壊れない（復旧できる）← **最優先**
- **Gate 2**: 主要フローが通る
- **Gate 3**: 体験が良い（Webとして完成）
- **Gate 4**: 運用できる（引き継げる）

## 🚨 Gate 1: 壊れない（復旧できる）

**目的:** システムが壊れても気づけて、すぐに戻せる状態にする

### 1.1 データ整合性とセキュリティ

#### データバリデーション
- [ ] **フロントエンドとバックエンドの両方でバリデーション**
  - [ ] 参加表明フォーム（会場/配信/両方の選択）
  - [ ] 応援メッセージ投稿（文字数制限、禁止ワード）

  - [ ] プロフィール編集（ユーザー名、表示名、性別）

- [ ] **不正なデータが保存されないか確認**
  - [ ] 負の数値が入力できないか
  - [ ] 空文字列が保存されないか
  - [ ] 不正なHTMLタグが保存されないか
  - [ ] SQLインジェクション対策（tRPCとDrizzle ORMで対応済み）

#### 認証・認可
- [ ] **ログイン状態の確認**
  - [ ] OAuthのコールバック保護（触らない方針維持）
  - [ ] セッション切れ/キャンセル/失敗が必ず自前UIで着地
  - [ ] ログイン不要で見れる範囲・必要な範囲が一貫

- [ ] **セッション管理**
  - [ ] セッションの有効期限が適切か
  - [ ] ログアウト時にセッションが破棄されるか
  - [ ] 複数デバイスでのログイン状態が正常か

- [ ] **CSRF対策**
  - [ ] tRPCのCSRF保護が有効か
  - [ ] 重要なアクション（参加表明、メッセージ投稿）にCSRFトークンがあるか

- [ ] **XSS対策**
  - [ ] ユーザー入力が適切にエスケープされているか
  - [ ] `dangerouslySetInnerHTML`を使用していないか

#### データベース
- [ ] **インデックスが適切に設定されているか**
  - [ ] `challenges`テーブル: `hostId`, `eventDate`, `createdAt`
  - [ ] `participations`テーブル: `challengeId`, `userId`
  - [ ] `messages`テーブル: `challengeId`, `userId`, `createdAt`

- [ ] **N+1問題がないか**
  - [ ] チャレンジ一覧取得時に参加者数を効率的に取得
  - [ ] 応援メッセージ一覧取得時にユーザー情報を効率的に取得

- [ ] **トランザクション処理が適切か**
  - [ ] 参加表明時に`currentValue`の更新が原子的に行われるか
  - [ ] 目標達成時の紙吹雪演出トリガーが正確か

#### レート制限/スパム対策
- [ ] **参加表明のレート制限**
  - [ ] 同一ユーザーが同じチャレンジに複数回参加できないか
  - [ ] 短時間に大量の参加表明ができないか（IP/ユーザー単位）

- [ ] **メッセージ投稿のレート制限**
  - [ ] 短時間に大量のメッセージを投稿できないか（IP/ユーザー単位）
  - [ ] 同じ内容のメッセージを連続投稿できないか

- [ ] **招待リンクのレート制限**
  - [ ] 短時間に大量の招待リンクを生成できないか

- [ ] **冪等性 or 二重防止**
  - [ ] 参加表明の二重送信を防ぐ仕組みがあるか
  - [ ] メッセージ投稿の二重送信を防ぐ仕組みがあるか
