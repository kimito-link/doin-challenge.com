# プロジェクト管理システム - テンプレート

**このテンプレートは、他のプロジェクトでも適用できる汎用的なプロジェクト管理システムです。**

「何回もやった作業をやらせる」「どこかなおしたらどこか崩れる」という問題を解決するため、以下の3つのドキュメントを作成し、プロジェクトの状態を一元管理します。

---

## 📋 システムの構成

このシステムは、以下の3つのドキュメントで構成されています：

### 1. `docs/status.md` - 状態管理

プロジェクトの現在の状態を一元管理します。

**目的**:
- AIと人間の両方が現在の状態を正確に把握できる
- 作業の重複を防ぐ
- 未解決の問題を明確に管理する

**更新タイミング**:
- 環境変数を追加・変更した場合
- 外部サービスの設定を変更した場合
- データベースのスキーマを変更した場合
- 新しい機能を実装した場合
- 未解決の問題が解決した場合
- 新しい問題が発生した場合
- チェックポイントを作成した場合

### 2. `docs/checklist.md` - チェックリスト

すべてのチェック項目を一箇所に統合します。

**目的**:
- プロジェクトの完成度を一目で確認できる
- 未実装の機能を明確に把握できる
- チーム間で進捗を共有できる

**更新タイミング**:
- 新しい機能を実装した場合
- チェック項目を完了した場合

### 3. `docs/README.md` - ドキュメント目次

すべてのドキュメントの目次を作成します。

**目的**:
- 必要な情報を素早く見つけられる
- ドキュメントの全体像を把握できる
- 新しいメンバーがプロジェクトを理解しやすくなる

**更新タイミング**:
- 新しいドキュメントを作成した場合
- ドキュメントの構成を変更した場合

## 📝 使用方法

### 1. プロジェクト開始時

1. `docs/status.md`を作成し、初期状態を記録
2. `docs/checklist.md`を作成し、実装すべき機能をリストアップ
3. `docs/README.md`を作成し、ドキュメントの目次を作成

### 2. 開発中

1. 機能を実装したら、`docs/status.md`を更新
2. チェック項目を完了したら、`docs/checklist.md`を更新
3. 新しいドキュメントを作成したら、`docs/README.md`を更新

### 3. 問題が発生した場合

1. `docs/status.md`の「未解決の問題」セクションに追加
2. 調査結果を記録
3. 次のアクションを明記

### 4. 問題が解決した場合

1. `docs/status.md`の「未解決の問題」セクションから削除
2. 「実装済み機能」セクションに追加
3. 更新履歴を追加

## 💡 ベストプラクティス

### 1. 更新は小まめに行う

作業が完了したら、すぐに`docs/status.md`を更新してください。

### 2. 正確な情報を記載する

不確かな情報は「⚠️ 未確認」と明記してください。

### 3. 日付とバージョンを記載する

更新履歴には必ず日付とバージョンを記載してください。

### 4. チェックポイントを作成する

`docs/status.md`を更新した後は、チェックポイントを作成してください。

### 5. AIに相談する

複雑な問題が発生した場合は、AIに相談しながら進めてください。

## 🎯 期待される効果

このシステムを導入することで、以下の効果が期待できます：

### 1. 作業の重複を防ぐ

AIが過去の作業を正確に把握できるため、同じ作業を繰り返すことがなくなります。

### 2. 現在の状態を把握できる

人間が現在の状態を一目で確認できるため、プロジェクトの進捗を把握しやすくなります。

### 3. 問題の早期発見

未解決の問題を明確に管理できるため、問題の早期発見と解決が可能になります。

### 4. デグレードの防止

「どこかなおしたらどこか崩れる」問題を防ぐため、変更の影響範囲を明確に管理できます。

### 5. チーム間の情報共有

チームメンバーが現在の状態を共有できるため、コミュニケーションコストが削減されます。
